Codex Ideation: Trail Catalunya Race Intelligence

date: 2026-06-11 topic: codex-trail-catalunya-fresh-ideation mode: repo-grounded author marker: Codex separate opinion

Grounding Context

This fresh pass treats Trail Catalunya as two linked products: a mobile-first race discovery website and a machine-readable race intelligence layer. The existing app already has useful filters, grouped cards, URL search params, a Supabase-backed scraper, a weekly Edge Function, change auditing, and an in-site Claude assistant. The deployed MVP presents a dense filter/list/card surface with drive, distance, elevation, month, province, kids run, and unscheduled toggles.

The strongest opportunity is not another static calendar. The original Ultres Catalunya page remains a source calendar, while your app can become the trusted layer that cleans, explains, enriches, and exposes race knowledge. Current local data also shows the shape of the hard problem: many races remain TBD, drive times are incomplete in the static cache, price data is sparse, and the data model is factual but not yet expressive enough for questions like “scenic, close to Barcelona, and ideally butifarra at the finish.”

External grounding: MCP is an open protocol for connecting AI tools to data sources and actions, and the official MCP tool model exposes named tools with schemas. The original source calendar says it is updated as dates are confirmed, which reinforces the need for freshness/provenance rather than one-off scraping. Comparable race calendars emphasize filters, maps, and category browsing; Trail Catalunya can differentiate through local semantics and agent-friendly explanations.

My opinionated frame: build one shared “race intelligence core” first, then let the website, chat assistant, JSON/API, and MCP server all call the same search, explain, compare, and recommend primitives. That keeps the product coherent and avoids four subtly different versions of truth.

1. Race Intelligence Core

confidence 92%complexity Medium
Axis: MCP and agent interface
Description: Create a shared internal module that owns race search, filtering, ranking, comparison, and explanation. The website filters, in-site assistant, public JSON/API, and future MCP server all call this same core rather than each reimplementing logic.
Basis: direct: the app already duplicates filter concepts between RaceList.jsx and app/api/chat/route.js; the chat route builds its own search_races tool over data/races.json, while the page now fetches grouped rows from Supabase.
Rationale: This is the highest-leverage move because “MCP-friendly” becomes a product architecture, not a bolt-on. If the same core can answer “races under 90 minutes, 15-25 km, scenic if known,” then UI chips, chat answers, and MCP tools stay consistent.
Downsides: It is less flashy than a map or new filter. It requires discipline around input/output schemas and ranking explanations.

2. Trust Layer: Freshness, Provenance, Confidence

confidence 90%complexity Medium
Axis: Data ingestion and trust
Description: Give every race and enriched field a visible source, last-checked time, confidence level, and “unknown vs not found” distinction. Surface this lightly on the website and fully through MCP.
Basis: direct: the scraper already writes scrape_runs and race_changes; the footer already displays last updated; current data has sparse fields such as prices and incomplete drive times.
Rationale: Trust is the product moat. If an agent recommends a race because it found “kids run” or “butifarra,” it must be able to say whether that came from Ultres, the organizer site, a cached page, or manual tagging.
Downsides: Adds metadata weight. The UI must avoid making every card feel like an audit log.

3. Taste Tags and Community Semantics

confidence 86%complexity Medium
Axis: Enrichment semantics
Description: Add a controlled vocabulary for qualities runners actually ask about: scenic, coastal, forest, high mountain, runnable, technical, vertical, family-friendly, beginner-friendly, festive, food at finish, butifarra, train-accessible, and “good first race.”
Basis: direct: your prompt names the impossible current query: scenic, not far from Barcelona, preferably with butifarra at the end. Existing structured fields only cover date, distance, elevation, province, drive time, kids run, status, and URL.
Rationale: This is where Trail Catalunya becomes community-native. Distance and elevation answer “can I physically do it?” Taste tags answer “will I enjoy this day?”
Downsides: Tags can become subjective or messy. Start with evidence-backed tags and allow lower-confidence “inferred” tags instead of pretending all tags are equally factual.

4. MCP Surface as a First-Class Product

confidence 84%complexity Medium
Axis: MCP and agent interface
Description: Design a small MCP server with resources for dataset snapshots and tools like search_races, compare_races, recommend_races, explain_race, get_race_changes, and fetch_source_evidence.
Basis: external: the MCP specification frames tools as schema-described capabilities exposed to models; your current chat assistant already has Anthropic-style tools for search_races and fetch_race_url.
Rationale: Do not make MCP a dump of races.json. Make it the same product promise as the website: “help me choose.” Good MCP tools return ranked candidates, field provenance, and concise explanation, not just rows.
Downsides: Public MCP hosting, rate limits, and prompt-injection hygiene matter once tools fetch organizer websites.

5. Enrichment Pipeline for Organizer Sites

confidence 80%complexity High
Axis: Data ingestion and trust
Description: Add a second-stage crawler that visits each race URL, extracts candidate facts, stores raw evidence snippets, and promotes only validated fields into the canonical dataset. Start with low-risk fields: registration status, price, GPX/course links, start time, kids races, food/after-party mentions, and official contact links.
Basis: direct: the existing chat route has a fetch_race_url tool and Supabase cache, but it is on-demand and text-extraction is simple regex/HTML stripping. The scraper already has weekly infrastructure and change logging.
Rationale: This unlocks the user’s “butifarra” class of questions. It also turns race pages into durable structured knowledge instead of one-off LLM browsing.
Downsides: Organizer websites are heterogeneous and brittle. This needs a quarantine/review model and should not overwrite canonical facts blindly.

6. Discovery Modes Beyond Filters

confidence 78%complexity Medium
Axis: Race discovery UX
Description: Keep the current filter list, but add opinionated modes: “near Barcelona this month,” “first trail race,” “family day,” “scenic 15-25 km,” “big mountain day,” “ultra training block,” and “make a weekend of it.” Each mode is just a saved query plus ranking explanation.
Basis: external: comparable race calendars expose filters and categories, but your differentiator is local route-selection judgement. direct: the existing UI is already dense enough that adding more raw chips may make mobile discovery worse.
Rationale: People often do not know the exact filter combination they need. Modes let the product express community taste without hiding the underlying filters.
Downsides: Requires careful curation so modes do not feel arbitrary. Best once taste tags and confidence exist.

7. Change Feed and Race Watch

confidence 74%complexity Low
Axis: Community utility
Description: Add a “what changed” view: new races, dates confirmed, price/status changes, sold out, removed, and newly enriched fields. Later, let users or agents watch a query like “Barcelona under 1h, 10-20 km, kids run.”
Basis: direct: race_changes already exists as an append-only audit log with public read policy. The scraper computes added, changed, removed, and re-added events.
Rationale: Community runners do not only browse once; they come back as registrations open and dates solidify. This is a natural MCP use case too: “what changed since last week?”
Downsides: Notifications can wait. The first version should just be a read-only page/resource.
