Trail Catalunya Reshape — Scraper, Filters & Agent Access

date: 2026-06-11 topic: trail-catalunya-reshape mode: repo-grounded

Focus: reshape the project into a best-in-class niche tool for the Catalunya trail-running community — the website with rich "cool" filters, the scraping pipeline behind it, and MCP access so anyone can use it via Claude or ChatGPT. The north-star query, in the maintainer's words: "a very scenic route not far from Barcelona, and if possible with butifarra at the end."

Grounding Context

Codebase context. Next.js 16 site on Vercel reading from Supabase. A weekly Deno Edge Function scrapes the ultrescatalunya.com calendar (Mon 05:00 UTC via pg_cron), parses ~600 race rows, diff-tracks every change into a race_changes audit table, and auto-redeploys the site via a Vercel hook when data actually changed. Current filters: distance, elevation, drive time from Plaça Glòries (pre-computed per town, 311/312 towns covered), month, province, kids run. Race metadata is sparse — name, date, km, D+, price, town, province, status, URL. Each race links to its own website containing rich prose: course descriptions, photos, historical results, aid-station and food details. A Claude chat widget exists in the codebase but is parked.

External context. No major platform — ITRA, UTMB Index, Betrail, Ahotu, DUV — offers drive-time proximity, qualitative race character, natural-language querying, or historical results merged into discovery. ITRA's commercial gating disadvantages small regional races, leaving the long tail open for a community tool. LLM enrichment of ~300 race websites costs roughly $3–30 per season; the bottleneck is anti-bot exposure, not money. Closest prior art: FindMyMarathon's derived course-character scores, Wikiloc's TrailRank (engagement as quality signal — built in Girona, used by exactly this community), LiteAPI's hotel "vibe search" (embed prose, match semantically), Google Data Commons' MCP pattern (abstract the schema behind natural-language-shaped tools), Surfline/eBird condition-triggered alerts, and published research on climbing-crag recommenders (33 qualitative attributes; users most wanted the attributes systems lacked — "beautiful views"). The MCP registry (~16K servers) has an underrepresented sports/events category.

Past learnings. No docs/solutions/ knowledge base exists yet; the project's durable decisions (event-as-atom grouping, geocode/drive-time cache separation, status semantics) live in CLAUDE.md prose and should be carried forward through any reshape.

1. The Race Dossier — an LLM-enriched qualitative layer

A1 · data-enrichment Confidence 90% Complexity Medium
Description
Add a second pipeline stage: for each race, fetch its own website (the URL is already in every row), and have an LLM extract a versioned dossier against a controlled vocabulary — terrain character (ridge / forest / coastal / volcanic), technicality, % runnable, atmosphere (village festa vs. competitive), post-race food (butifarra detection, literally), village size, shade/exposure — plus a 2-sentence "character note" stored as a pgvector embedding. Store the raw scraped prose corpus alongside the extracted fields so future extraction passes can re-run without re-scraping. Re-enrich only when the weekly diff detects the race site changed. All six ideation frames independently arrived at this idea — the strongest convergence signal in the run.
Basis
directThe dream query — "scenic… with butifarra at the end" — is unanswerable from the current schema; race URLs with rich prose already sit in every row. externalLLM enrichment of ~300 sites ≈ $3–30/season; LiteAPI's vibe search proves the embed-prose pattern; crag-recommender research catalogued 33 qualitative attributes as a workable schema shape.
Rationale
Every other ambition — semantic search, good MCP answers, qualitative filters, alerts on "scenic races" — is gated on this data existing. Nobody in the trail space has built it, and at this cost it fits a hobby budget. The controlled vocabulary is the key design decision: fixed descriptors work simultaneously as filter chips, URL params, MCP tool enums, and embedding fodder.
Downsides
LLM extraction reads race marketing, which is uniformly positive ("every race claims to be scenic") and goes stale — needs idea 6's verification loop eventually. Anti-bot friction on some race sites. The attribute list is a one-shot schema decision that everything else inherits; getting it wrong is expensive.

2. One Query Engine — natural language that compiles to inspectable filters

A2 · discovery-ux Confidence 85% Complexity Medium
Description
A natural-language box above the existing filters. "Scenic ridge run under 90 min away in May, food at the end" is compiled by one cheap LLM call into (a) hard structured constraints — drive time, month, distance — rendered as the existing filter chips so the user sees and can adjust what the machine understood, and (b) a semantic rerank over the dossier embeddings for the qualitative terms, with a one-line "why this matched" per result. The origin becomes a parameter too: pick your town once (Girona, Vic, Manresa…) instead of everything being hardwired to Plaça Glòries — required anyway for agents whose users could be anywhere. Filters stay for power users; nothing is taken away.
Basis
directThe dream query is literally a natural-language sentence; the parked chat widget signals the appetite existed before the data was ready. externalResearch confirms no major platform has NL querying; pgvector ships free in the Supabase the project already runs.
Rationale
Filters force runners to translate desire into facets; most can't express "scenic" that way. Compiling to visible chips avoids the black-box trust problem that kills most NL search UIs. And the same compiler is exactly the contract the MCP server needs — one engine serving the site, the agents, and (later) alert matching.
Downsides
Hard dependency on idea 1's dossier layer for the qualitative half. Per-query LLM cost needs the precompute-heavy design (embeddings at scrape time, cheap cached parse at query time). Mixed-language queries (Catalan/Spanish/English) need care.

3. MCP Server with intent-shaped tools — the site becomes a client

A3 · agent-access Confidence 85% Complexity Medium
Description
Ship a remote MCP server (Supabase Edge Function or Cloudflare Worker, streamable HTTP) whose tools do not mirror the database schema. Instead: search_races(query, hard_filters), get_race_dossier(id) — which returns the structured tags plus the raw prose corpus, so the calling agent can do its own butifarra reasoning at query time — compare_races(ids), whats_on(date_range, origin, max_drive), and whats_changed(since). Responses are compact, citation-bearing, and carry freshness timestamps. Refactor the Next.js site onto the same Supabase RPCs so site and MCP are two clients of one canonical query layer, and the parked chat widget revives for free. Add llms.txt and JSON-LD; register in the public MCP directory.
Basis
directThe user's stated requirement: "MCP-friendly so other people can use it via Claude or via ChatGPT." externalData Commons MCP pattern (abstract the schema; agents shouldn't learn table layouts); the MCP registry's sports/events category is underrepresented — early-mover visibility is real.
Rationale
A schema-mirroring API makes every agent re-derive domain logic (event grouping, TBD semantics, null-elevation handling) and most will get it wrong; intent-shaped tools encode that logic once, server-side. The dossier tool is the hedge: even before the controlled vocabulary is complete, handing agents the raw prose lets a frontier model answer qualitative questions today. Making the site a client of the same layer means every future field ships to both surfaces for free.
Downsides
A public tool contract is hard to change once agents depend on it — the tool surface deserves real design time. Unauthenticated public endpoints need rate limiting. Site refactor onto shared RPCs is real work beyond the server itself.

4. Race Radar — saved-criteria alerts on the existing diff stream

A5 · community-retention Confidence 80% Complexity Small–Medium
Description
Let visitors save a standing query — structured ("new race <1h from BCN, 20–30k, 1000+ D+") or, once ideas 1–2 land, semantic ("scenic, with a festa") — with just an email or a calendar-subscription token, no account. Each weekly scrape's diff is matched against saved criteria and fires notifications: new race matching your taste, date changed, price posted, registration opened, sold out, cancellation. Agents can manage the same subscriptions through MCP tools (watch_criteria, get_my_matches) so "have my Claude watch the calendar for me" becomes a real workflow.
Basis
directThe race_changes table already diff-tracks every row weekly with before/after snapshots — the hard half (change detection) is built and currently feeds nothing but redeploys. externalSurfline's condition alerts and eBird's rarity alerts are the canonical proof that change-triggered notifications, not the reference data itself, are what turn niche outdoor sites into daily-relationship products.
Rationale
A discovery site gets visited twice a season; an alert service earns a permanent slot in someone's inbox. This is the one feature where the weekly-cron architecture is an advantage rather than a limitation, and it creates the project's first user relationship (an opt-in list) at near-zero marginal pipeline cost.
Downsides
First feature requiring stored user data on a deliberately zero-auth architecture — privacy posture and email infrastructure (deliverability, unsubscribes) need deliberate choices. Weekly cadence caps alert freshness; truly time-critical events (flash sell-outs) would need idea 7's registration-platform sources.

5. Race Biography — editions, results, and real difficulty

A1 · data-enrichment Confidence 65% Complexity High
Description
Stop treating "La Maasaiada 2026" as the atom. Model the durable entity — the race — with editions per year, linked by URL/name continuity, and join historical results where they exist (race-site results pages; DUV for ultras). Derive empirical character per race: median finish pace vs. flat-equivalent expectation, DNF rate, field size, finisher-count trend, cutoff pressure, consecutive-years-running. Surface as a "reality check" on race cards ("small and brutal" vs. "big and runnable") and as a filterable difficulty independent of the organizer's self-description. The existing race_changes audit log becomes longitudinal memory — date drift, price history, cancellation reliability — instead of a write-only table.
Basis
externalFindMyMarathon's PR Score validates deriving course character from results — absent in trail everywhere; DUV holds 10M historical performances with zero discovery UX on top. reasonedkm and D+ wildly under-describe trail difficulty (a technical 20k can outrank a runnable 35k); finish-time distributions are the one attribute organizers cannot embellish, making this the ground-truth anchor that calibrates idea 1's softer extracted attributes.
Rationale
Answers the second-most-natural runner question after "where" — "can I actually finish this?" Results compound season over season into the dataset no calendar competitor has, and the entity model unlocks year-over-year features ("alert me when my race opens next year") that the flat calendar can't express.
Downsides
Results scraping is a second, messier pipeline — PDFs, chip-timing platforms, heterogeneous Catalan formats — plus personal-data considerations around named finishers. The cross-season identity key should be decided before the 2027 scrape starts, even if results ingestion comes later.

6. Verified Vibes — triangulating prose, geometry, and finishers

A5 · community-retention Confidence 60% Complexity Medium
Description
Make the qualitative layer trustworthy by triangulating three independent signals. (1) Prose claims — idea 1's LLM extraction from the race's own website. (2) Geometry confirms — computed badges from open geodata: course GPX or start-town corridor against ICGC land cover, DEM profiles, distance-to-coast, Parc Natural overlap, photo density from Wikiloc/Flickr ("ridge-line %", "sea views likely"). (3) Finishers verify — the Monday after each race, a 15-second anonymous micro-survey linked from the race page and alert digest (scenery 1–5, marking, what was the food?). Three concordant reports flip a tag from LLM-inferred to community-confirmed, displayed as a confidence badge; questions are generated from dossier gaps ("we don't know if this race has butifarra — did it?").
Basis
externalWikiloc — founded in Girona, used by this exact community — proved Catalans contribute structured outdoor data at volume; crag-recommender research showed implicit signals can seed qualitative scoring before explicit reviews exist; Komoot computes tour character from geodata rather than prose. reasonedRace marketing is uniformly positive, so prose alone is a corrupted signal for exactly the attributes that matter most; "was the butifarra actually there" is knowable only by attendees, and the race calendar tells you precisely when and whom to ask.
Rationale
This is the difference between "AI-summarized" and "community-verified" — a moat no scraper-only competitor can copy, and the mechanism that keeps idea 1's data honest and self-improving each season. It also converts users from readers into stakeholders, which is what "community tool" actually means.
Downsides
First write path into a zero-backend-philosophy project: moderation, spam, rate limiting, anonymity-vs-identity all need decisions. Cold-start before survey volume exists (mitigated by the geodata leg). GPX availability is uneven across small races.

7. Multi-Source Pipeline — entity resolution over several calendars

A4 · pipeline-integrity Confidence 70% Complexity High
Description
Treat ultrescatalunya.com as one feed among several. Add scrapers for the FEEC federation calendar, ITRA's Catalunya listings, and — highest value — registration platforms (Inscripcions.cat, RockTheSport), which uniquely yield price tiers, registration open/close dates, and live sold-out status. Behind them, an entity-resolution layer (fuzzy name + date + town matching) merges records into one canonical race with per-field source precedence and provenance. Disagreements between sources surface as flagged diffs ("date disputed") rather than silent rot, and the pipeline survives any single source dying or changing markup. The first extra source is expensive (you build the resolver); every subsequent one is cheap.
Basis
directThe entire dataset currently hangs off one weekly scrape of one site — both a coverage ceiling and an existential fragility. reasonedA "best-in-class" trust claim collapses the first time a runner finds a missing race or drives to a moved one; registration status and pricing — the two fields with real-world urgency — are structurally unavailable from a calendar-only source.
Rationale
Moves the project from "mirror of one website" to "the canonical Catalunya race dataset" — which is also what makes the MCP server trustworthy enough for agents to cite, and what gives idea 4's alerts something time-critical to fire on.
Downsides
Entity resolution across messy Catalan race names is the riskiest engineering bet on this list, with ongoing per-source maintenance. Registration platforms may have stronger anti-bot. Defer until ideas 1–4 prove the product, but design the race-identity key (idea 5) compatibly now.
